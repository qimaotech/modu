## Context

在 TUI 界面的操作菜单（menu）和列表视图（list）中，用户经常需要获取当前选中项的主项目路径。当前只能通过"打开 VS Code"（`o`）间接实现，不够直观。

## Goals / Non-Goals

**Goals:**

- 在操作菜单中增加"复制路径"菜单项（位置：o 之后，u 之前）
- 在列表视图中支持按 `c` 直接复制路径
- 复制成功后显示临时消息 "路径已复制: <path>"
- 使用跨平台剪贴板库支持 macOS 和 Linux

**Non-Goals:**

- 不支持复制相对路径
- 不支持复制其他模块路径（仅主项目路径）
- 不需要用户确认流程

## Decisions

### 1. 快捷键选择

**决定**: 使用 `c` 作为快捷键

**理由**: `c` 在菜单中尚未被占用，且与 "Copy" 首字母一致，直观易记。

### 2. 菜单项位置

**决定**: 插入到位置 1（`o` 之后，`u` 之前）

**理由**: 用户心智模型中，"打开"和"复制"都是获取路径的方式，逻辑相近。`u`（更新代码）属于不同类别的操作。

### 3. 剪贴板库选择

**决定**: `github.com/atotto/clipboard`

**理由**:

- 零依赖，封装了各平台的原生方案
- macOS: `pbcopy`
- Linux: `xclip` 或 `xsel`
- 纯 Go 实现，无需 CGO

### 4. 路径来源

**决定**: 与 VS Code 打开逻辑完全一致

- 主项目: `mainProject.Path`
- Feature: `env.MainProject.Path`

**理由**: feature worktree 本身不是独立目录，其路径就是主项目路径。

### 5. 错误处理

**决定**: 如果 `env.MainProject` 为 nil，显示错误消息（与 `o` 操作一致）

**理由**: 保持行为一致性，避免用户困惑。

## Risks / Trade-offs

| 风险 | 说明 | 缓解 |
|------|------|------|
| 剪贴板工具缺失 | Linux 系统可能没有 `xclip`/`xsel` | atotto/clipboard 会返回错误，显示错误消息即可 |
| 路径包含特殊字符 | 空格、中文等可能影响终端使用 | 剪贴板复制是纯文本，不受影响 |
